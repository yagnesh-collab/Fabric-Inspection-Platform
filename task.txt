Following our interview, we’d like to invite you to the next stage of our hiring process: a technical take-home assignment. The goal is to understand how you approach system design, architectural decisions, and code quality in a problem similar to our production environment.

To give you some context, here’s a simplified overview of how our system works:

Our Jetson Orin devices run an anomaly detection model on images coming from circular knitting machines. The data sent to the cloud falls into two categories:

Whenever a new anomaly class is detected, the corresponding frame is uploaded to the cloud for future retraining.
Any anomaly detection result whose confidence exceeds a predefined threshold is also sent to the cloud as an alarm/event.
Jetson telemetry and machine sensor data, on the other hand, can be stored locally either on disk or via MQTT. We’re interested in which approach you would choose and, more importantly, why.

We’d like you to complete the following two-part assignment:

1. Cloud Side
Build a queue/consumer service that receives the two event types described above:

New anomaly class detected
Threshold exceeded
The service should process incoming events and store them in a persistence layer of your choice. It should also expose a simple metrics/health endpoint showing information such as:

Total number of processed events
Current queue backlog
Event distribution by type
Any other metrics you consider useful
The choice of queue/broker is entirely up to you (Redis, RabbitMQ, LocalStack/SQS, or any other solution). Just include a short explanation in the README describing why you chose it.

Note: No cloud infrastructure (AWS, GCP, Kubernetes, etc.) is required. Everything should run locally using Docker Compose.

2. Jetson Simulation + Monitoring UI
For this part, we’d like you to simulate a real Jetson device.

Please use a publicly available fabric/textile defect dataset (Kaggle has several suitable options). Stream the images sequentially as if they were coming from a live industrial camera. You may use WebSockets, MJPEG, or any streaming approach you prefer.

The UI should include:

A live camera feed displaying the images sequentially
A mocked anomaly detection result for each frame (e.g. anomaly class and confidence score)
A visual alert whenever the configured threshold is exceeded
A panel showing the cloud ingestion/health status using the endpoint implemented in Part 1
Short Design Notes
Please also include a brief document covering:

Your choice between local disk storage and MQTT, along with your reasoning.
What considerations you would make before taking this architecture to production (roughly half a page is sufficient).
Finally, we’d like to emphasize that the requirements above are intended as a guideline rather than a strict specification. If you’d like to introduce your own ideas whether that’s an alternative architecture, additional functionality, or different tooling—we’d be happy to see them.

Likewise, if you believe a particular requirement is unnecessary or could be improved, feel free to modify it. We simply ask that you briefly explain your reasoning in the README, as we’d like to discuss these design decisions together during the follow-up interview.

We are thinking of approximately four days for the assignment, but this is not a strict deadline—we’re happy to be flexible based on your availability.

Once you’re finished, simply send us the repository link.

If you have any questions or would like to discuss any ideas during the process, feel free to reach out at any time.
